# 第 3–4 周工作计划（四人并行，弱对接）

## 1. 本阶段在项目里的位置

第 3–4 周视为**一个连续冲刺**，**截止时间：第 4 周最后一天**。  
本阶段的目标是：在不大规模互相等待的前提下，把**大部分可独立落地的代码与文档骨架**做出来；**前后端联调、RAG 与编排的完整串起来、端到端演示**，安排在更后面的周次。

当前仓库状态：后端已有 FastAPI 骨架与健康检查；前端为 Vue 3 + TypeScript 单页演示；MySQL / Redis 通过 Docker 提供。业务上的「任务、生成、大纲结构、检索证据」等尚未实现，适合本阶段四人各守一条主线并行推进。

---

## 2. 并行原则（为什么可以「少对接」也能交卷）

- **接口先写成「契约」再写实现**：谁依赖 HTTP，就只认「请求长什么样、响应长什么样、错误码长什么样」，不依赖对方代码是否已合并。
- **用 Mock（假数据）顶住 UI 与评测**：前端和测试同学用本地 JSON / 小型 Mock 服务即可把页面与脚本跑通；等后面周次再换成真实后端。
- **模块边界清晰**：检索、编排、页面展示、评测各自落在不同目录；合并冲突时优先各写各的文件，少改公共入口文件。
- **本阶段不要求**：四人代码在同一分支上「一次跑通全流程」；不要求真实大模型与向量库在生产环境就绪（可用占位实现 + 可切换的 fake 模式）。

## 3. 组长（A）需要盯住的清单

「全组最小公约数」已落盘（组长维护，后续有变更请同步版本号并通知全组）：

- **《API 与数据结构契约（v0）》**：[docs/api_contract_v0.md](../api_contract_v0.md)（当前 **v0.1.0**）。包含：`POST /api/tasks` 创建任务、`GET /api/tasks/{task_id}` 查询状态与结果快照、`PATCH .../clarification` 提交或更新澄清、`POST .../generate` 触发生成；任务状态 `TaskStatus`、错误体 `error.code`、`Outline` 与 `evidence_catalog` 字段说明；以及供 B 对齐的 `RetrievalHit` 内存形状。

以下 B、C、D 的任务描述里，凡写「按契约」均指 [docs/api_contract_v0.md](../api_contract_v0.md)。

## 4. B（RAG 检索负责人）——第 3–4 周主责

**目标一句话**：把「能从资料里捞出相关片段、能控制深度、能把证据结构化交出去」做成**可运行的 Python 模块**，即使暂时不接真实 HTTP 也没关系。

建议工作拆成可独立完成的小块（顺序可自定，但都在本阶段内闭环）：

- **检索策略与数据源抽象**：定义「一份输入查询 + 可选过滤条件 → 若干条检索结果」的函数或类接口；数据源先用本地文件夹、少量 Markdown / PDF 样例即可，不强制接线上语料。
- **向量检索 / 重排占位**：若环境或密钥未就绪，允许先用「关键词匹配 + 固定排序」或「随机但结构正确」的 fake 实现，但接口形状要与真实实现一致，便于下周替换。
- **深度档位 L0 / L1 / L2 与真实性规则**：用配置或枚举表达不同档位下：召回条数上限、是否启用重排、是否必须带来源文件名与页码/段落偏移等；把规则写成代码常量 + 简短说明文档（放在 `docs/` 或模块内 `README` 片段均可）。
- **证据回填结构**：输出里固定包含：`snippet`（片段文本）、`source_id`、`score`（可选）、`locator`（页码或偏移）、`confidence`（可选）等字段，**字段名与 A 的契约对齐**；若契约尚未细化，B 可先定一版草案并在组内频道留言，由 A 在 v0 里吸收或标注「待定」。
- **工程化习惯（备份职责顺带做）**：为检索调用预留超时、重试、简单缓存的钩子（可先空实现或 TODO，但接口留好）。

**本阶段交付物（自检用）**：一个可被 `pytest` 或简单 `if __name__ == "__main__"` 脚本调用的检索包；附带几段示例调用与期望输出形状说明。

## 5. C（前端交互负责人）——第 3–4 周主责

**目标一句话**：把「需求澄清 → 任务状态 → 大纲结果展示」三条用户路径做成**完整可用的前端界面**，数据来自 Mock 即可，字段严格按 A 的 v0 契约。

建议工作拆成可独立完成的小块：

- **路由与页面骨架**：至少三个视图：需求澄清表单页、任务进度 / 状态页、结果展示页（章节 / 页面 / 要点 / 证据 / 备注分区展示，可先静态样式为主）。
- **Mock 数据层**：用 `frontend/src` 下的 JSON 或小型 `mockApi` 函数模拟异步请求与轮询，**不要等待后端真实接口**；切换开关留好（例如环境变量或常量：`USE_MOCK = true`），便于后面一周改成真实 `fetch`。
- **类型与字段对齐**：为契约中的核心类型写 TypeScript 类型定义（`interface` / `type`），页面只消费这些类型，避免魔法字符串散落。
- **交互细节**：表单校验、加载中、失败重试、空状态提示；任务状态用标签或步骤条表达，枚举值与 A 文档一致。
- **备份职责（轻量）**：对照契约检查「若后端按此返回 JSON，前端是否每个字段都有展示位」；记录一页「联调准备清单」（本阶段不要求真联调，只列出将来要替换的接口路径与字段）。

**本阶段交付物（自检用）**：`npm run dev` 下可完整点完三条路径；Mock 数据与类型文件齐全；截图或简短录屏可选但不必强求。

## 6. D（测试评估与文档负责人）——第 3–4 周主责

**目标一句话**：把「怎么评、评什么、结果怎么记」落成**可执行的评测脚本骨架 + 指标口径文档**，不依赖真实服务全通。

建议工作拆成可独立完成的小块：

- **指标口径文档**：用通俗语言定义：成功率、平均响应时间（若暂不可测则写「占位」）、大纲完整性（例如必填字段是否齐全）、证据覆盖率（有出处的要点占比）等；每个指标写清：分子分母、合格线（可先草案）、数据来源（日志 / JSON 文件 / 人工表）。
- **评测数据集 v0**：小规模即可（例如 5–20 条），每条包含：用户输入主题、期望的大纲层级深度、是否必须带引用等；存成 JSON 或 CSV，路径与命名规范写进文档。
- **自动化评测骨架**：用 Python 或 Node 写「读入数据集 → 调一个可替换的 `generate_outline(...)` 桩函数 → 输出评分结果到 `reports/`」；桩函数可先返回静态 JSON 或读本地样例，与 B、C 无运行时依赖。
- **人工评测记录模板**：用分级标题与列表即可（例如每条样例下列：评分人、分数、问题描述、是否阻塞发布）；避免在正文用大段表格，方便在 Git 里做 diff。
- **备份职责（轻量）**：列一份「后续与真实 API 对接时要测的用例清单」（等价于测试用例 backlog）。

**本阶段交付物（自检用）**：跑一条命令能生成一份评测报告目录（哪怕全是占位分数）；指标口径文档可被新手读懂并照着重测。

## 7. A（组长，后端编排）——第 3–4 周主责

**目标一句话**：把「任务与生成流程」在后端落成**清晰的模块划分 + 路由壳 + 内存或 DB 持久化占位**，不要求第 4 周最后一天就接好大模型与 RAG，但**接口形状稳定、可被 Mock 消费**。

建议工作拆成可独立完成的小块：

- **领域模型与存储占位**：任务实体（id、状态、创建时间、输入快照、输出快照指针等）；先用内存字典或 SQLite / MySQL 任一最小实现均可，关键是状态机能跑通。
- **路由与用例**：按 v0 契约实现或**先实现「返回契约示例的 stub 路由」**再逐步填逻辑；确保 `/docs` 里能看到新接口，方便 C、D 将来对齐。
- **编排占位**：用异步任务（后台线程、FastAPI `BackgroundTasks`、或 Celery 占位均可）模拟长时间生成；返回 `task_id` 与轮询查询接口。
- **配置与安全底线**：API Key、模型端点等从环境变量读取，代码里不写死密钥；日志里打任务 id，不打用户隐私全文（可脱敏）。
- **组内推进（主责）**：第 3 周前半锁定 v0 契约；第 4 周中段组织一次短会（可选）只做「契约变更收束」，避免各人理解漂移。

**本阶段交付物（自检用）**：后端能通过 Postman / Swagger 走完「创建任务 → 查询状态 → 拿到 stub 结果」；契约文档与代码字段一致。
